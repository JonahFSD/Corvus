## 3. Reality recon — VERIFIED landmarks (2026-06-14; re-verify before citing)

Coordinates confirmed on disk. `path` is relative to the `OpenFoundry/` repo root. Line numbers are approximate anchors — `rg` to confirm.

**Ontology data model (the schema gold — this is the single most valuable file)**
- `services/ontology-service/migrations/20260419100004_initial_ontology.sql` — the **whole** ontology schema in ~70 lines. Five tables: `object_types` (L3: `primary_key_property`, `owner_id`, icon/color), `properties` (L16: `property_type TEXT`, `required`, `unique_constraint`, `default_value JSONB`, **`validation_rules JSONB`** — note: validation is a per-property JSONB blob, **there is no reusable `value_type` table**), `link_types` (L32: `source_type_id`/`target_type_id`, **`cardinality TEXT DEFAULT 'many_to_many'`** — stored but **not** enforced by any constraint), `object_instances` (L46: `object_type_id` + **`properties JSONB NOT NULL DEFAULT '{}'`** + `created_by` — a hybrid system-columns-plus-JSONB row, **confirming our blueprint's D5**), `link_instances` (L55: a **unified edge table** `link_type_id`/`source_object_id`/`target_object_id` + `properties JSONB`, **confirming our blueprint's D6**). Everything is `ON DELETE CASCADE`.
- `services/ontology-service/src/models/*.rs` — domain types: `object_type.rs`, `property.rs`, `link_type.rs`, `action_type.rs` (real); **`constraint.rs` and `interface.rs` are empty (0-byte) stubs** — value-type constraints and interfaces are *unimplemented*. `domain/type_system.rs` (~47 lines) holds `validate_property_type` / `validate_property_value`.
- `services/ontology-service/src/handlers/objects.rs` (~127 lines) — object CRUD over `object_instances`. `domain/{graph,indexer,sync,time_series}.rs` and `domain/search/*` and `handlers/{search,bulk}.rs` are **all 0-byte stubs** — traversal, search, indexing, bulk are scaffolding only.
- `libs/core-models/src/{ids,error,pagination,timestamp}.rs` — tiny shared types. **There is no `dataset`/`transaction`/`security` module and no version/transaction model** (compare to our D2: OpenFoundry has *no* dataset-transaction concept to harvest).
- `proto/ontology/{ontology.proto, object.proto, action.proto}` — the typed contracts (the "Protobuf as source of truth" our brainstorm noted; `proto/` + `buf.gen.yaml` drive codegen). Read for *shape/vocabulary*; the codegen pipeline itself is platform-era (§2 reject).

**Action / edit / audit model (the kinetic core)**
- `services/ontology-service/migrations/20260423113000_action_types.sql` — **one** table, `action_types` (`object_type_id`, `operation_kind`, **`input_schema JSONB`**, **`config JSONB`**, `confirmation_required`, `permission_key`, `owner_id`). **There is no `action_log`, `action_execution`, `revert`, or `what_if`/`scenario` table anywhere** (the v1 prompt invented them — `rg -il 'action_log|action_execution|revert|scenario' --type sql` returns nothing).
- `services/ontology-service/src/handlers/actions.rs` (**~1200 lines — substantial and real**) — the heart. `validate_action_definition` (~L437), `validate_action` (~L959) returns an `ActionPlan` + errors (**server-side validation, confirming the submission-criteria concept**); `execute_action` (~L993) builds `ActionPlan::{UpdateObject, CreateObject, DeleteObject, CreateLink, …}` and applies them with `sqlx` (`INSERT`/`UPDATE`/`DELETE FROM object_instances` at ~L1095/L1175). Side-effects are **HTTP invocation** declared in the action `config` (`HttpInvocationConfig` struct L56, `validate_http_invocation_config` L234) — **not** a transactional outbox.
- **Critical gap to confirm:** `execute_action` does plain writes with **no transaction/isolation, no `version` guard, and `rg -n 'audit|emit|AuditEvent' services/ontology-service/src` returns nothing** — i.e. the action path appears to **write objects without recording any edit/audit event**, even though the service *does* depend on `event-bus` + `auth-middleware` (`services/ontology-service/Cargo.toml`). Verify this absence yourself; if confirmed it means OpenFoundry's kinetic core **does not implement the edit-history MUST** — our `edit_event` design is *ahead* of theirs.

**Audit & identity (B5)**
- `services/audit-service/src/models/audit_event.rs` — a genuinely good, harvestable shape: `AuditEvent` (~L79) is **hash-chained** (`sequence` L81, `previous_hash` L82, `entry_hash` L83 — tamper-evident), with `actor` (String, L86), `action`, `resource_type`/`resource_id`, `status`, `severity`, `classification`, `metadata`, `occurred_at`. `domain/immutable_log.rs` builds the chain. Note it's a **separate service**, decoupled from the write path (see the gap above).
- `libs/auth-middleware/src/{claims.rs, jwt.rs, rbac.rs, row_level_security.rs, tenant.rs}` — identity flows as **JWT claims** → `RlsContext` (`row_level_security.rs:9`: `user_id`, `org_id`, `roles`, `permissions`, `attributes`). This is the `actor` provenance to map onto **Supabase Auth** (`auth.uid()` / JWT claims).

**Expression / rules language (B1)**
- There is **no embedded expression DSL** for actions. Action rules are a **structured JSON `config`** (mappings like `property_name`, `target_object_id`, create/update specs) validated by **hardcoded Rust** (`type_system::validate_property_value`). No CEL, no Pratt parser, no type lattice (`rg -il 'cel|pratt|lexer'` → nothing relevant).
- Pipeline transforms live in `services/pipeline-service/src/domain/engine/mod.rs` — a node runs a **`transform_type`** of `sql` / `python` (`pyo3`) / `wasm` (`wasmtime`), *not* a typed expression language. So: OpenFoundry has **no solution to harvest for B1** — it punts exactly where we must decide. That *confirms* B1 is genuinely ours to resolve (our CEL leaning stands on its own; OpenFoundry neither helps nor refutes it).

**Authorization (B6 / security-travels-with-data)**
- No Cedar. Real authz: `libs/auth-middleware/src/{rbac.rs, row_level_security.rs}` + `services/auth-service/src/domain/{rbac.rs, abac.rs}`, `models/policy.rs`, `handlers/policy_mgmt.rs`. The intended model is RBAC + ABAC + an `RlsContext` to "scope DB queries to the user's org/permissions."
- **But the Hasura-style "compile policy into the `WHERE`" is unbuilt:** `libs/query-engine/src/optimizer_rules.rs:3` literally reads `//! Future: Row-level security filter injection`. So our brainstorm's "compile predicates into the read query" leaning is **aspired-to but not implemented** in OpenFoundry — compare against **Supabase RLS** (real, in-DB, enforced on reads) as the lean alternative.

**No worked example, and immature docs**
- **There is no `docs/` directory** and no `poc-online-retail` / `use-case-development` material. `ARCHITECTURE.md`, `CHANGELOG.md`, `CONTRIBUTING.md`, `SECURITY.md` are **all empty (0 bytes)**; only `README.md` and `ROADMAP.md` have substance (and they describe *intent*, much of which is stubbed in code — see the 0-byte files above). ## 4. Reconcile against our six blockers (the core of the harvest)

For **each** blocker, write: *what OpenFoundry actually does* (`path:line`, VERIFIED vs REPORTED), *whether it confirms / improves on / contradicts our brainstorm leaning*, and *the final lean answer for the wedge*. Expected findings from the recon above — confirm or overturn each:

- **B1 — expression/rules language:** OpenFoundry has **none** for actions (JSON `config` + hardcoded Rust validators) and uses SQL/Python/WASM for pipelines. It offers no DSL to copy → our **CEL** leaning is unrefuted and stands. Recommend CEL for criteria/policies/value-constraints, or — for *one* workflow — possibly even simpler (a handful of typed TS predicates).
- **B2 — ingestion vs edits-only:** look for a datasource-vs-edit *merge*. The recon suggests OpenFoundry **doesn't have one** — `object_instances.properties` is a single JSONB blob written directly by actions; ingestion (`data-connector`/`dataset-service`) is separate. If confirmed, OpenFoundry does **not** validate our two-layer `datasource ⊕ edit` model (that idea comes from Foundry-proper's docs, per our brainstorm, not from OpenFoundry). Say so plainly.
- **B3 — concurrency:** confirm there is **no** `version` column on `object_instances` and **no** isolation/locking/optimistic guard in `execute_action`. OpenFoundry has the *same* write-skew exposure our audit reproduced → it confirms the gap, offers no fix. Our **SERIALIZABLE + `version` guard** leaning stands.
- **B5 — identity→actor:** harvest the real shape — JWT claims (`auth-middleware/claims.rs`) → `actor` String stamped on `audit_event` (`audit_event.rs:86`). Map to **Supabase Auth** (server-derived `auth.uid()`, never client-supplied). This *confirms* our B5 direction and hands us a concrete field shape.
- **B7 — ontology change once data exists:** look for any migration/versioning that re-validates existing `object_instances` against changed `properties.validation_rules`. The recon suggests **none** (schema is `CREATE TABLE IF NOT EXISTS`; no ontology-version table). If confirmed, OpenFoundry gives **no** answer here → B7 remains ours (the deepest hole), and the two-layer model still helps as our brainstorm argued.
- **A6 — soft-delete vs audit:** OpenFoundry **hard-deletes** (`DELETE FROM object_instances`, `ON DELETE CASCADE` throughout) — it does the *exact thing our audit warned against* and keeps no tombstone. This **contradicts** our plan only in that OpenFoundry is *worse*; it confirms *by counterexample* that our **tombstone + delete-is-an-event** design is the right call. Name it as such.

> The through-line: on most blockers OpenFoundry is **not ahead of us** — it either punts (B1, B7), shares our gap (B3), or is actively worse (A6, and the un-wired audit). The genuine harvest is narrower and more concrete than v1 implied: the **schema shapes** (objects-as-JSONB, unified edge table, per-property `validation_rules`, the hash-chained `audit_event`, the `action_types` `input_schema`/`config` split) and the **identity→actor** wiring. Lead with that honesty.

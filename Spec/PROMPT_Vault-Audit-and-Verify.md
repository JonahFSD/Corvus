# Prompt — Obsidian Vault Audit & Verify (Claude Code CLI / dev-tools)

> **How to use:** run in a **Claude Code CLI** session with `python`/`node`/`git`/`rg` from a directory that can see the vault. **Set `$VAULT`** to the vault root (the folder containing `.obsidian`). Two candidates live here:
> - `Foundry/Vault` — 73 notes, a dense research graph (~1,946 wikilinks), **generated by `Foundry/build_vault.py`** — read the wrinkle in §0.
> - `OBSIDIAN` — 9 notes, a hand-authored engineering-principles vault.
>
> **Spirit:** *an audit that is itself unverified is just another opinion.* Every structural finding is the output of a script you actually ran; every content claim is marked VERIFIED / REPORTED; every finding cites `note:line`. Do **not** trust the vault's self-description, or this prompt's appendix, or a prior recon — derive the vault's conventions from its own files. (This prompt exists because a sister prompt's "verified landmarks" were written from assumption and turned out to be fiction. Don't repeat it.)

---

## 0. The first question: is this vault *generated* or *authored*?

Answer it before anything else, because it changes what "fix" means.

- Look for a generator (`rg -l "open\\(.*w" --type py` near a `Vault/` write; a `build_*.py`; templated/identical headers; "generated — do not edit" markers; note prose that also appears as string literals in a script).
- **`Foundry/Vault` is generated.** `Foundry/build_vault.py` (~1,459 lines) holds the note content **inlined as Python dicts** and writes the `.md` files out. **Therefore the source of truth is the generator, not the notes** — any edit you make to a `.md` is erased on the next run. For a generated vault, **every fix targets `build_vault.py` (and any inputs it reads), and the audit says so explicitly.** Auditing the rendered `.md` is still how you *find* defects; the generator is where you *fix* them.
- `OBSIDIAN` is hand-authored — fixes go in the notes themselves.

Emit, as the first artifact, a one-screen **VAULT PROFILE**: generated-vs-authored (+ generator path), note count, attachment count, the frontmatter schema actually in use, the link/alias conventions, and the MOC/hub structure. Everything downstream is calibrated to this profile — not to assumptions about "how Obsidian vaults usually look."

## 1. Profile the conventions (assume nothing)

- **Frontmatter schema:** derive it from the corpus (`rg` the YAML blocks, tally keys + types). If the vault ships its **own** standard — here `Foundry/Vault/00 Maps & Views/Vault Metadata Standard.md` — treat that as the contract and audit conformance **to it**, rather than inventing a schema.
- **Link conventions:** `[[wikilink]]`, `[[link|alias]]`, `[[link#heading]]`, `[[link^block]]`, `![[embed]]`, and plain Markdown links. **Collect every declared `aliases:` value — including YAML block-list form spread over multiple lines — so link resolution is alias-aware.** (A naive checker that ignores aliases will cry wolf; see the appendix.)
- **Reachability model:** find the MOCs / hubs / index notes and how a reader is meant to reach every note (here: numbered domain folders + invariant/layer hubs).

## 2. Structural integrity — deterministic (write a script; the output is evidence, not opinion)

Produce `check_vault.py` (or `.ts`) that parses every note **once** and reports, each finding with `note:line`, a proposed fix, and an `auto-fixable?` flag:

- **Broken internal links & embeds** — resolve `[[…]]` / `![[…]]` against basenames, relative paths, **and declared aliases**; report the unresolved.
- **Orphans** — zero inbound links **and** absent from every MOC/index (a note reachable only by folder browsing is still an orphan in graph terms).
- **Attachments** — embedded/linked files that don't exist; and files in the vault referenced by nothing.
- **Frontmatter** — YAML parses; required keys present (per the vault's own standard if it has one); no duplicate keys; types sane; dangling/empty values.
- **Name & alias collisions** — two notes resolving to the same name or alias (silently misdirects links).
- **Tags** — casing drift, near-duplicates, singletons.
- **Detritus** — committed test/scratch files (here: `Foundry/Vault/.junk/…`), `.DS_Store`, empty or stub notes (< N words), TODO/`FIXME`/`XXX` placeholders left in prose.
- **(opt-in) external link rot** — HTTP(S) HEAD with backoff; off by default (slow, network).

`check_vault.py` is itself a deliverable: a **re-runnable gate**, not a one-shot pass — in the spirit of the `Deterministic Gates` note in the sibling `OBSIDIAN` vault. Wire it so a human (or CI) can run it after every `build_vault.py` change.

## 3. Content truth — semantic (the expensive pass: judgment, not regex)

This is the half a script can't do, and the half that matters most for a *research* vault whose notes assert facts about the world.

- **Claim ledger.** For each substantive note, extract its nontrivial factual claims and mark each **VERIFIED** (a cited source exists **and actually supports the claim**) / **REPORTED** (a source asserts it, unchecked) / **UNVERIFIED** (no source) / **CONTRADICTED**. A citation that *resolves* is not a citation that *supports* — check support.
- **Cross-layer consistency.** A generated research vault sits in a provenance chain: deep-research → `ResearchResults/` → `build_vault.py` → notes → `Spec/`. Flag where a note **contradicts its own source**, contradicts a **sibling note**, or contradicts the **resolved Spec** (the Spec is the later, audited decision record — if a note still preaches something the Spec/Audit overturned, that's drift, not truth).
- **Staleness.** Dated or "current as of" claims, product/version specifics, anything that rots. For external facts, prefer a fresh check over the note's say-so.

Scope the semantic pass deliberately (it's costly): do the hubs + invariants first, then the highest-inbound notes, then the long tail.

## 4. Coherence & graph health

- **Merge candidates** — near-duplicate notes covering one idea (note-consolidator territory).
- **Coverage gaps** — MOCs that promise notes which don't exist; concepts linked but never authored. *(On this vault, `[[Action Parameter]]` and `[[Submission Criteria]]` are referenced as first-class concepts yet appear to lack their own notes — confirm; if real, they're coverage gaps, not typos.)*
- **Reachability** — every note reachable from a hub; hubs neither bloated nor starved.

## 5. Deliverables (write under `$VAULT/_audit/` or `Foundry/Harvest/`)

1. **`VAULT_AUDIT.md`** — exec summary + the VAULT PROFILE + findings ranked **P0** (broken/contradicted/data-loss) → **P3** (nits), each with `note:line`. Lead with the punchline: the few things that actually undermine trust in the vault.
2. **`structural.md`** — the §2 tables (the script's output).
3. **`claims.md`** — the §3 ledger + the contradiction list.
4. **`fixes.md`** — a prioritized plan splitting **auto-fixable** (exact edits or a script) from **needs-judgment**. **For `Foundry/Vault`, every fix is phrased as a change to `build_vault.py`** (with the dict/line to touch), never to the rendered `.md`.
5. **`check_vault.py`** — the re-runnable structural gate from §2.

## 6. Discipline (the non-negotiables)

- **Reality-first.** §0–§1 derive conventions from the files. Never assume a schema or link style.
- **Deterministic vs judgment, kept separate.** Structural facts are script output you ran — quote the command. Content claims carry VERIFIED/REPORTED and name the supporting source.
- **Generated-vault rule.** If a generator exists, audit the rendered notes to *find* defects but fix the *generator*; state plainly that note edits are ephemeral.
- **Alias-aware, block-list-aware resolution** — or your broken-link and orphan counts are wrong (proven below).
- **Never auto-delete or mass-rewrite.** Propose fixes; work on a git branch; keep the diff small and reviewable; let a human approve deletions and merges.
- **The audit must itself be verifiable.** Cite `note:line`; show the `rg`/script you ran. Don't let the audit become another unverified recon — that failure mode is the whole reason this prompt exists.

# fixes.md — prioritized remediation plan

Work on a branch; keep diffs small; let a human approve deletions. Each fix is routed to the **correct pipeline stage** — which, for this vault, is usually **not** `build_vault.py`. Read the routing model first; it overturns the audit prompt's "always fix the generator" premise.

---

## Where fixes go (the corrected operating model)

The audit prompt asserts: *"the source of truth is the generator … any edit you make to a `.md` is erased on the next run … every fix targets `build_vault.py`."* **That is false for this vault** (proved in `VAULT_AUDIT.md` P0-1, and conceded by `README.md:184`). The real routing:

| If the defect is in… | Source of truth | Where to fix | Safe to regen? |
|---|---|---|---|
| Prose/links in a **consolidated doc** (03–10) | the **rendered `.md`** | edit the `.md` directly | ❌ `build()` would resurrect atomic stubs |
| Prose/links in an **atomic note** (02 Foundations) | the **rendered `.md`** (hand-rewritten links) | edit the `.md` directly | ❌ `build()` would revert link rewrites |
| **Section footers / inline `#tags`** (04–10) | `vault_tools.py normalize` | re-run `normalize --apply` (idempotent) | ✅ |
| **Build Matrix** counts | `vault_tools.py views` ← `build_vault.CONCEPTS` | edit `clone=`/layer default in `build_vault.py`, then `views --apply` | ✅ |
| **Build View / hubs / Home / Taxonomy / Design Decisions** | emitted by `build_vault.build()` **but frozen** (unsafe to re-run) | edit the `.md` directly **and** mirror into `build_vault.py` data | ⚠️ only via full pipeline rebuild |
| **Concept registry / dispositions** | `build_vault.py CONCEPTS` | edit `build_vault.py` (canonical data, consumed by `views`) | n/a |

**Rule of thumb:** `vault_tools.py` is safe and idempotent; `build_vault.py build()` is a one-way scaffold that has already been superseded by consolidation. Treat `build_vault.py` as the canonical **concept data file**, not as a renderer you re-run.

## P0 — stop the foot-gun (data-loss prevention)

### F0-1 · Make the regeneration hazard impossible to trip *(needs judgment; high value)*
The README and this audit's prompt both invite `python3 build_vault.py`, which would corrupt the vault. Do all three:

1. **Guard the README.** `README.md:178-184` "Regenerating the vault" currently buries the warning. Replace the bare `python3 build_vault.py` with a prominent banner: *"⚠️ Do NOT run this against the live vault — it reverts consolidation. See pipeline below."* and document the full safe pipeline.
2. **Guard the script.** Add an opt-in flag to `build_vault.py` so a naked run refuses:
   ```python
   if __name__ == "__main__":
       import sys
       if "--atomic-into" not in sys.argv:   # require an explicit empty target dir
           sys.exit("Refusing to write atomic notes over the consolidated vault. "
                    "Pass --atomic-into <empty-dir> to regenerate the atomic layer.")
   ```
   (Or point `VAULT` at a scratch dir unless overridden.) This is the single most important fix.
3. **Make consolidation reproducible.** Commit the real group-configs for folders **03, 05, 06, 07, 08, 09, 10** (only `assets/example-groups.json`, a 3-doc sample, exists) and the link-rewriting step. Until then, the consolidated vault cannot be rebuilt — so the rendered `.md` files are load-bearing originals and must be backed up / treated as source.

*Auto-fixable?* No — these are design decisions. But #2 is a ~5-line change.

It has 5 `##` sections but no frontmatter `tags:` and no inline section tags, so `vault_tools.py normalize` (which only rewrites *existing* `*clone:*` footers) skips it permanently and tag/graph queries miss it.
- Quick fix: add a frontmatter `tags:` block mirroring its scalars — `tags: [layer/semantic, kind/service, clone/drop]` (it's all-drop) — so the whole doc is at least file-level queryable. *(edit the `.md`)*
- Thorough fix: add the per-section `*clone: …*` footers (all `drop`) so `normalize` adopts it thereafter, OR teach `vault_tools.py normalize` to seed footers on `##` sections that lack them. *(edit `vault_tools.py`)*

### F2-2 · Resolve the folder-03 disposition-flattening *(needs judgment)*
`Dataset & Transactions`, `Data Connection (Magritte)`, `Data Health & Lineage`, `Transforms & Build` each bundle mixed dispositions under one file-level `clone`, so `#clone/<x>` filters mislead and the Standard's own carrier rule is violated.
- Option A (cheapest): demote these from carrier-A to **carrier-B** (give each `##` section + `*clone:*` footer like 04–10), so per-concept disposition is tag-queryable. Then `vault_tools.py normalize` maintains them — but note `normalize`'s `FOLDER_LAYER` (`vault_tools.py:31`) covers only 04–10; **add `"03 Data Foundation": "data"`** to bring 03 under the normalizer.
- Option B: keep file-level `clone`, but pick the value honestly (a doc with any `drop` member shouldn't tag `keep`) and drop the inconsistent inline `*Clone: X*` markers. Loses per-concept queryability but stops the active mislead.
- The Build Matrix already reflects the correct per-concept dispositions, so this is a note-tag fix, not a data fix.

## Optional structural improvement (removes a whole class of drift)

### F-extra · Make the Build View generated like the Build Matrix
Both views derive from `build_vault.CONCEPTS`, but only the Build Matrix is emitted by the **safe** `vault_tools.py views`. The Build View is emitted by the **frozen** `build_vault.build()`, which is why it drifted. Move Build View generation into `vault_tools.py views` (it already imports `build_vault` and knows `disposition()`), rendering the consolidated `[[Doc#Section|…]]` links from the concept→doc map. Then both views self-heal on `views --apply` and can never disagree again. *(edit `vault_tools.py`; needs judgment)*

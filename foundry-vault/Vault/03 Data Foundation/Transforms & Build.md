# Transforms & Build

> Authoring → orchestration: declarative transforms (code or no-code) registered in pipelines, run as a build DAG, kept fresh by schedules, and made efficient by incrementality.

**Transform.** A declarative mapping from input datasets to output datasets, bundling compute logic and resources. Authored in code or generated by Pipeline Builder; registered in a pipeline; run by the build system; can be incremental. Produces the curated datasets that [[Objects#Object Backing|back objects]].

**Code Repositories.** Git-backed authoring for code transforms (Python/PySpark/Java/SQL); committing publishes **JobSpecs**, and Preview runs on sampled inputs without committing. The code-first counterpart to Pipeline Builder.

**Pipeline Builder.** A no-/low-code, point-and-click, type-safe graph that compiles to a general transform backend (Spark, Flink, or the Rust/DataFusion engine — see [[Compute Backends & Economics]]). Outputs can be datasets, [[Objects#Object Type|object types]], links, streams, or exports.

**Build System.** The data-driven orchestrator (internally 'Build2') that computes new dataset versions: it builds a DAG of **jobs** from JobSpecs, skips 'fresh' outputs (unchanged inputs + logic), and supports retries/aborts. Driven by schedules.

**Schedule & Trigger.** Runs builds over time (cron) or in response to events (an input dataset updating) — time-based, event/transaction-based, and streaming triggers. Project-scoped is recommended over user-scoped.

**Incremental Computation.** Process only what changed — and decide that automatically by inspecting input transaction types. The `@incremental` decorator reads build history and runs incrementally iff inputs only had files added ([[Dataset & Transactions|APPEND]]); a SNAPSHOT input forces a full rebuild. The keystone efficiency mechanism. *Clone: keep the concept; trivial at small scale.*
